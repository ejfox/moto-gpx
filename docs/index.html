moto-gpx

A zero-dependency Node.js CLI that turns a folder of GPX tracks into a stack of map-ready GeoJSON layers — tracks, stops, speed bins, photos, weather, OSM roads and places, state crossings, SRTM hillshade, OSRM route comparisons, sun position. Built for hand-making dope QGIS maps of multi-day motorcycle trips.

github.com/ejfox/moto-gpx  ·  releases  ·  MIT

What it does

Given a folder of .gpx files — phone, Garmin, helmet cam, whatever — moto-gpx walks recursively, merges every track point into one sorted timeline, dedupes, splits on break-gaps to find stages, and emits a stack of layered GeoJSON ready to drag straight into QGIS. Every .geojson gets a sibling .qml style file so the symbology auto-loads.

Optional enrichments add weather (Open-Meteo), OSM roads and place names (Overpass), OSRM route comparisons (what the car would've done), state/province crossings, sun position on each photo, and a seamless SRTM DEM + hillshade (AWS Terrain Tiles + GDAL). A exiftool-backed --media flag places photos and videos directly from EXIF GPS or interpolates them along the track by timestamp.

Every run ends with a superlatives banner — the year-in-review reading for a single ride.

Install

git clone https://github.com/ejfox/moto-gpx.git
cd moto-gpx && npm link

Needs Node 18+. Optional: exiftool for --media, GDAL for --dem. Zero npm install — the tool is stdlib + fetch only.

Run

moto-gpx ~/trips/big-sur \
  --media ~/trips/big-sur \
  --enrich all \
  --dem --dem-contour 100 \
  --out ./big-sur-out

Then drop ./big-sur-out/ into QGIS. Every layer comes pre-styled.

Output

out/
├── all.geojson            every stage as a LineString
├── stages/                one LineString per break-split stage
├── days/                  one file per calendar day
├── days-merged/           ONE unbroken LineString per day
├── hours/                 per-hour slices
├── stops.geojson          rest / long-rest / overnight Points
├── speedbins.geojson      line chunks tagged slow|moderate|fast|highway
├── markers.geojson        stage + day start/end Points, pre-labeled
├── media.geojson          photos + videos placed on the map
├── crossings.geojson      state/province transition Points
├── places.geojson         towns along the route (osm)
├── roads.geojson          OSM ways you rode, by highway class (osm)
├── pois.geojson           viewpoints, peaks, fuel, historic (osm)
├── optimal_routes.geojson  OSRM "what you should've done" (routes)
├── weather_timeline.json  hourly weather per stage region (weather)
├── dem/                   SRTM tiles + VRT + hillshade + contours (dem)
├── styles/                full QGIS .qml templates
└── stats.json             totals, per-stage breakdown, superlatives

CRS is plain WGS 84 (EPSG:4326). See LAYERS.md for every property on every layer.

Design

  • Zero runtime dependencies. Stdlib + global fetch only. No npm install needed to run. exiftool and GDAL are the only external binaries, and only when the features that need them are enabled.
  • Everything opt-in except the tracks. A default run hits zero external APIs. Every enrichment is a flag.
  • Fail soft. One flaky API never aborts a run. Retries once, then logs and skips. Missing GDAL preserves raw DEM tiles so you can stitch them by hand.
  • WGS 84 throughout. Drop into QGIS and go — no reprojection.
  • QGIS-native output. .qml siblings auto-load on drag. Zero symbology configuration to get a readable map.
  • Cache what's expensive. DEM tiles persist in dem/tiles/. Re-running on the same bbox is free.
